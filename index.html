docker搭建pytorch cuda版

docker搭建pytorch cuda版

本地主机配置

保证本地主机有如下三个配置:

  • nvidia显卡驱动
  • cuda toolkit(貌似可以不装, NVIDIA Container Toolkit 官网介绍–Quickstart第2行有提到)
  • cudnn

这些都是操作系统之上调用硬件GPU所必须的配置

Docker container配置

  1. 将pytorch装在docker container中,若想让pytorch调用本地GPU加速,需要有cuda toolkit和cudnn包. 通过pytorch安装命令conda install pytorch torchvision cudatoolkit=10.2在anaconda中安装时,可以观察到cuda toolkit和cudnn包被安装了.

    **注意: **此处的cuda toolkit和cudnn包相当于接口, 与本地主机配置中的不一样. 但要保证两者的版本相同.

  2. 在Dockerfile中,直接使用pytorch官方镜像pytorch/pytorch:1.6.0-cuda10.1-cudnn7-runtime进行配置, 在container中pytorch所依赖的一切都是配好了的.

本地主机和Docker container的衔接

  1. 在docker 19.03版本之前, 需要在本地主机上安装插件nvidia-docker2, docker容器内的pytorch必须通过此插件才能调用本地主机的GPU资源.

  2. 从docker 19.03版本开始, nvidia的GPU能自动支持容器中应用的调用. 详情请参考: NVIDIA Container Toolkit官方文档

  3. 如NVIDIA Container Toolkit官方文档所说, docker container和host os之间需要添加中间件: nvidia-container-toolkit,添加方法见官方文档!

  4. 中间在网站上搜索了众多关于如何在docker中搭建pytorch cuda版博客, 均介绍需要添加中间件nvidia-container-runtime,添加方法见nvidia-container-runtime官方文档. 为避免出错, 将3和4来两个中间件都加上.

    添加nvidia-container-runtime时, 先按照nvidia-container-runtime官方文档添加repository, 然后apt-get install nvidia-container-runtime即可.

  5. 在添加nvidia-container-toolkit 和 nvidia-container-runtime时可能出现错误no valid openPGP data found, 建议先卸载安装了的nvidia-container-toolkit和nvidia-container-runtime(如果有的话), 然后unset http_proxy,unset https_proxy,重启再重新添加. 参见no valid openPGP data found 错误.

在如上三个配置都OK的情况下, 使用如下Dockerfile制定一个专属镜像,其中包含miniconda, pandas, scikit-learn 和jupyterlab.

  1. 使用命令docker run -it -p 9991:8888 --name pytorch-sj-test2 --restart=always --gpus all --mount type=bind,source=yourpath,target=/workspace monkeytao/pytorch:1.6.0-cuda10.1-cudnn7-runtime bash创建容器, 其中:
    • pytorch-sj-test2是容器名,需自己另设

ubuntu系统迁移

Ubuntu系统迁移

写作时间：2020年7月29日

环境

Machine OS
Origin Machine1 ubuntu1
Target Machine2 ubuntu2

迁移方向：Origin –> Target

引导方式：UEFI + GPT

前话

  1. 因为ubuntu一切皆文件, 故迁移系统就是将ubuntu2的系统文件（/目录下所有文件）完全替换成ubuntu1。
  2. 系统迁移和备份是同一个原理——都是拷贝复制OS文件， 但两者的环境略有不同。
    • 系统备份：它是在本电脑上恢复OS文件，硬盘分区和引导位置都未改变；
    • 系统迁移：既然是迁移，那么就是不同电脑之间的操作，硬件环境一定发生了变化，只讲OS文件替换进去是不行的，还要配置环境。

UEFI + GPT 系统引导程序那些事儿

  1. ubuntu系统就是/目录下的所有文件，它存放于硬盘GPT分区的某个位置上。电脑开机的时候，不会智能到自动找到该OS文件并执行的地步，需要引导程序告诉计算机在什么位置寻找OS，故引导程序实际就是一道桥梁的作用。

  2. 对于一个在UEFI模式下安装的ubuntu系统，在安装时，某硬盘（本文是sda）一定有如下两个分区：

    • /dev/sda1—>OS System：此分区存放Ubuntu的系统文件，其中包含：

      • /etc/fstab, /etc/fstab.d（可能没有）：启动文件
      • /boot/grub/grub.cfg：分区文件

      注：在fstab启动文件中，可以看到里面根据分区的UUID绑定了/dev/sda1给/，将/dev/sda2绑定到/boot/efi中，这里也是启动的关键。如果UUID不正确，就无法找到引导程序和OS文件，系统恢复后无法正常开机。

    • /dev/sda2 —> EFI System：此分区是在EFI下安装Ubuntu必要的，里面应该存放了引导程序。

  3. 系统备份与迁移

    • 系统备份：因为备份恢复前后，电脑的分区不会改变，而且备份恢复改变的只是OS所在分区，不会影响到/dev/sda2的EFI System。所以就算换了OS文件，系统还是能正常引导修复。因此，系统备份没有系统迁移复杂，只需要复制拷贝OS文件即可。

    • 系统迁移：因为是将系统从一台电脑移动到另外一台电脑，硬盘分区会变，EFI文件也会改变。所以对于系统迁移，建议：

      在machine2上先重装系统，新系统中EFI System、fstab启动文件以及grub.cfg分区文件都是匹配的。即，新系统下一切正常，只需要删除ubuntu2然后移入ubuntu1即可，其他不变。

系统迁移正文

事先准备一个ubuntu启动盘

系统备份In Machine1

  1. Machine1 —> U盘启动 —> Try Ubuntu进入livecd模式: 开终端，进入管理员权限。

  2. 挂载ubuntu1所在分区和移动硬盘

    • mkdir data：在/目录下，创建data。
    • mount /dev/sda1 /mnt：假设ubuntu1在Machine1的sda1上，将其挂载到mnt上
    • mount /dev/sdb1 /data：假设移动硬盘在Machine1的sdb1上，将其挂载到data上

  3. 备份ubuntu1至移动硬盘

    • 方法一：tar备份

    参数：
    -c： 新建一个备份文档
    -v： 显示详细信息
    -p： 保存权限，并应用到所有文件
    -z： 用gzip压缩备份文档，减小空间
    -f： 指定备份文件的路径
    –exclude： 排除指定目录，不进行备

    排除如下文件：
    /proc：一个虚拟文件系统，系统运行的每一个进程都会自动在这个目录下面创建一个进程目录。既然是系统自动创建，也就没必要备份的必要了。
    /tmp：一个临时文件夹，系统的一些临时文件会放在这里。
    /lost+found：系统发生错误时（比如非法关机），可以在这里找回一些丢失文件。
    /media：多媒体挂载点，像u盘、移动硬盘、windons分区等都会自动挂载到这个目录下。
    /mnt：临时挂载点，你可以自己挂载一些文件系统到这里。
    /run：系统从启动以来产生的一些信息文件。

    • 方法二，mksquashfs（如果用tar备份和恢复时出错，就用该方式）

      mksquashfs /mnt/data/ubuntu.sfs：备份系统

      sync：让系统保存下数据

  4. 解除绑定：

    umont /data

    umount /mnt

系统恢复 In Machine2

  1. 使用启动u盘在Machine2上以EFI模式安装ubuntu2系统，本文分区如下：

    • sda1 –> ubuntu
    • sda2 –> EFI System

  2. 安装新系统之后，依然从U盘启动，Try Ubuntu进入livecd模式: 开终端，进入管理员权限。

  3. 挂载：

    • mkdir data：在/目录下，创建data用以挂载移动硬盘
    • mount /dev/sda1 /mnt：将ubuntu2挂载到mnt
    • mount /dev/sdc1 /data：将移动硬盘挂载到/data

  4. 保留原ubuntu2的启动文件和分区文件：（如果是本地系统备份的话，可略此步）

    因为这两个文件记录了machine2上ubuntu2上的启动信息和分区信息，而系统备份中的文件是machine1的，放在新环境下，配置一定出错，所以要用本土的配置文件，只需换个系统文件即可。

    • cp /mnt/etc/fstab（若有fstab.d，就一并复制） /data：保留原启动文件
    • cp /mnt/boot/grub/grub.cfg /data：保留原分区文件

  5. 系统恢复

    • 方法一： tar

      rm -rf /mnt/*：将挂载到/mnt上的原系统彻底删除（可能此命令有误，可用其他命令）

      tar -xvpzf /data/ubuntu.backup /mnt：将备份系统恢复到挂载到/mnt上的ubuntu

      cp /data/fstab(fstab.d) /mnt/etc：恢复启动文件（本地备份恢复不需要此步）

      cp /data/grub.cfg /mnt/boot/grub：恢复分区文件（本地备份恢复不需要此步）

    • 方法二：mksquashfs

      mkdir /rescovery/mnt

      mount -o loop /data/ubuntu.sfs /rescovery/mnt：将备份文件恢复至/rescovery/mnt中

      cp -a /rescovery/mnt/* /mnt：将备份文件完整地拷贝至ubuntu2挂载的/mnt中

      cp /data/fstab(fstab.d) /mnt/etc：恢复启动文件（本地备份恢复不需要此步）

      cp /data/grub.cfg /mnt/boot/grub：恢复分区文件（本地备份恢复不需要此步）

    注：如果只是本地备份恢复到此即可结束（仅仅拷贝粘贴系统文件即可），如果是迁移系统，还需要有如下步骤！第6步，在系统迁移时试过了，但在系统启动时还是会出错；第7步，应该可行，但sudo add-apt-repository ppa:yannubuntu/boot-repair添加失败，来者需要特别注意。

  6. 引导修复1

    • 从try ubuntu中为ubuntu2挂载虚拟文件系统，为引导修复做准备：

      mount -o bind /dev /mnt/dev

      mount -o bind /proc /mnt/proc

      mount -o bind /sys /mnt/sys

    • chroot /mnt：进入ubuntu2

    • 对比UUID

      • 在try ubuntu中新建一个终端，输入blkid查看硬盘分区UUID

      • 在上面chroot /mnt过的终端中，使用vim /etc/fstab查看启动文件中的信息。对照/sda1（mount point为/）和/sda2（mount point为/boot/efi）绑定的两个UUID是否和/dev/sda1和/dev/sda2相吻合。（因为是拷贝的原ubuntu2系统中的文件，应该不会有错误，有错误的话，就需要修改。）

      • grub-install /dev/sda

Dockerfile用build定制镜像时多出来的镜像

Dockerfile用build定制镜像时多出来的镜像

写作时间:2020年8月4日

问题描述：在dockerhub中找到continuumio/anaconda3镜像的Dockerfile，本地build该Dockerfile后，发现本地镜像中不止会有anaconda3的镜像，还有很多none镜像，包括一个基础镜像。但是如果用docker pull continuumio/anaconda3:latest，拉取到本地的镜像是一个独立的文件。

前期思考：按照分层存储的概念，anaconda3镜像不应该是一个文件，而是以debian为基础镜像的多层文件联合组成。这样不将debian写死在anaconda3镜像中，有可能其他某个镜像也需要以debian为基础，那么该镜像和anaconda3可以共用debian（只需要一个debian），而如果将其写死，那么就需要两个debian，反而造成空间浪费。所以在本地build时，它不是一个独立的文件是好事。

猜想：dockerhub中下载的镜像（由基础镜像定制而来）是个独立的镜像，其中包含了基础镜像。即：如果有imageA&imageB，它们都由基础镜像image0构造而来，那么将imageA 和imageB pull到本地后，本地不会有image0,因为image0包含在imageA和imageB中。而如果用imageA和imageB的Dockerfile在本地定制镜像，那么本地会有基础镜像image0,且iamgeA和imageB中不包含image0,它们只是以image0为基础镜像所在的修改部分。

分析与思考

分析

  1. 为什么build Dockerfile的8个步骤中，除了最终的anaconda3:v1镜像外，还生成了7个其他的镜像?

    查阅资料知：

    • 在定制镜像时，每一个Dockerfile命令都会建立一层，因为有8个Dockerfile命令，所以依次建立了8层镜像。
    • 且，docker镜像采用Union FS分层存储，它不是一个文件，而是由一组文件系统组成，或者说，由多层文件系统联合组成。镜像构建时，都是一层一层地构建，前一层是后一层的基础。故，此例中，定制的目标镜像anaconda:v1是由这8个镜像一起构成的，且第一步命令提交的镜像是第一层，第二步提交的镜像是第二曾，依次类推。可以从build后的docker中的镜像列表观察到，由上至下的镜像顺序，正好是Dockerfile中命令提交镜像顺序的逆序。

  2. 在8条Dockerfile提交的镜像中，为什么有6个镜像只有ID ，没有具体的REPOSITORY和TAG呢？

    • 之所以第一步提交的ae8514941ea4镜像有debian:latest，是因为第一个Dockerfile命令是FROM debian:latest，实际在build过程中是从dockerhub中pull debian:latest，如图：

    • 之所以第八步提交的 cf4bb8f8044c镜像有anaconda:v1，是因为在build该Dockerfile时指定了tag为anaconda:v1，即最后的镜像就是anaconda:v1。从build过程最后两句”Successfully built cf4bb8f8044c”和”Successfully tagged anaconda3:v1”可知缘由。

    • 之所以中间几步提交的镜像没有具体的REPOSITORY和TAG，是因为没有指定。

  3. 那么，如果想将该anaconda:v1 push到dockerhub下的帐号中以供别人pull，应该怎么push？是只push anaconda3:v1呢，还是将其所依赖的其他7个镜像一并push?

思考

  1. 在问题描述中如果用docker pull continuumio/anaconda3:latest，拉取到本地的镜像是一个独立的文件。，这句话是错误的，continuumio/anaconda3:latest镜像看起来是个独立的文件，其实在pull的过程中，会pull多层文件，而不是一个单独的文件，如图所示：

    注：从pull镜像的过程中可以看出分层存储的概念，镜像是由多层存储所构成。pull也是一层层地pull，整个过程会pull多个文件下来，并非pull单一文件。pull过程中给出了每一层的 ID 的前 12 位。并且下载结束后，给出该镜像完整的 sha256 的摘要，以确保下载一致性。

  2. 定制完镜像push时，只需要push目标镜像，不用管底层所依赖的镜像。在pull该定制镜像时，和其他官方镜像的pull过程一样。注意分析部分最后两张关于push和pull过程的图片，由对应图片前面的ID知，push和pull的对象不是一个单一的文件，而是一层一层的多层文件。

收获

  1. 对docker分层存储这一卓越的做法，理解地更加深刻了。镜像由多层存储所构成，不要再将镜像想成是一个单一的文件，而是多层文件，且下一层构建在上一层之上。
  2. 人生苦短，多看官方文档！
  3. 最后，本文第一段话是解决问题之前的思考，现在看来很不成熟，漏洞颇多。之所以不删，是因为将整个完整的思考始末写下来，而不是直接抛出个最终解决方案，这样有利于从头到尾去思考最终方案是怎么来的，中间遇到过什么问题，为什么最后会想到这样的方案。一来方便自己回首反思，二来方便读者了解始末。

遗留问题：

  1. 在docker push monkeytao/anaconda3:v1之后，发现在自己帐号仓库中的镜像压缩大小为1.56G，但是continuumio/anaconda3:latest镜像大小仅为848.27MB，用的是同一份Dockerfile build的镜像，为什大小却不同呢？

  2. 在push定制的镜像时，因为镜像是多层存储所构成，所以push是一层层地在push，这一点可有push图片前面的镜像ID知道。但问题是，仔细对照push过程的镜像ID，发现没有哪一个ID是build过程提交的镜像ID，即在docker image ls -a的结果中，找不到这些镜像信息。那么这些镜像ID又究竟是怎么一回事呢？

  3. 在新机器上pull一个镜像后，通过docker image ls -a发现，在pull过程出现的几个镜像ID并未出现在本地的镜像列表中。那么同2问，这几个push过程中的镜像ID又是怎么回事呢？

    猜测，可能push和pull过程中行头出现的ID并不是镜像的ID，而是其他什么ID。docker在push时，会自动将目标镜像所依赖的底层文件打包进仓库，而在pull时，又会自动释放这些底层文件。
